## CUDA library setup

The installed `/usr/lib/python3.12/site-packages/nvidia/*` packages expose
CUDA 12 libraries, while the available ORT 1.27 CUDA provider requires
`libcublas.so.13`, `libcudart.so.13`, `libnvrtc.so.13`, and
`libcufft.so.12`. Initial runs silently executed on CPU (0% GPU SM activity)
and were discarded. Temporary CUDA 13 wheels were installed in the worktree:

```bash
python3 -m pip install --quiet --no-deps --target .bench-cu13 \
  'nvidia-cublas==13.0.2.14' \
  'nvidia-cuda-runtime==13.0.96' \
  'nvidia-cuda-nvrtc==13.0.88' \
  'nvidia-cufft==12.0.0.61'

export LD_LIBRARY_PATH=/usr/lib/python3.12/site-packages/nvidia/cublas/lib:/usr/lib/python3.12/site-packages/nvidia/cuda_nvrtc/lib:/usr/lib/python3.12/site-packages/nvidia/cudnn/lib:$LD_LIBRARY_PATH
export LD_LIBRARY_PATH=$PWD/.bench-cu13/nvidia/cu13/lib:/usr/lib/python3.12/site-packages/nvidia/curand/lib:/home/justinchu/onnx-genai/.ort-cuda-1.27/root/lib:$LD_LIBRARY_PATH
```

`ldd libonnxruntime_providers_cuda.so` then reported no missing dependencies.
The temporary wheel directory was removed after measurement.

## Performance

```bash

  --tokens TOKEN_COUNT \
  --prompt 'Explain why reproducible inference benchmarks matter in one concise paragraph.' \
  --warmups 1 --runs 5
```

`profile_decode` reports aggregate wall throughput over all five measured
runs, not a median distribution. It does not separately report prefill.

## Analysis versus 2026-07-13

The prior H200 report recorded 481.2 tok/s through HTTP at 1024 tokens and
~475 tok/s in the direct profiler with CUDA graph enabled. Wave 8 reaches
484.44 tok/s direct:

- **+2.0%** versus the prior direct-profiler checkpoint (~475 tok/s).
- **+0.7%** versus the prior HTTP number (481.2 tok/s), though direct engine
  and HTTP streaming are different measurement surfaces.
- Decode falls only **1.6%** from 128 to 1024 tokens (492.49 → 484.44 tok/s),
  so long-output scaling remains stable.
- Against the prior same-machine competitor numbers, Wave 8 remains about
  **+7.0%** over Foundry Local (452.6 tok/s) and **-4.5%** behind Ollama
  Q4_K_M (507.0 tok/s). Those competitor values were not rerun in this wave.

No performance regression is visible after the CUDA stack is correctly
loaded. The discarded apparent results (43.06 tok/s at 128 and 28.68 tok/s at
1024) were CPU fallback caused by unresolved CUDA 13 provider dependencies,
not valid GPU measurements.

## Harness improvements to consider

- `cargo build --release -p onnx-genai-bench` does not build
  `profile_decode`, and `bench-ort` does not forward CUDA; make the intended
  profiler build command discoverable or add a benchmark-level forwarding
  feature.
- Fail loudly when CUDA is requested but the CUDA provider cannot load or
  assigns no compute nodes. The current path can look successful while running
  on CPU.
- Report prefill separately and emit per-run samples/median/p90 rather than
  only aggregate wall throughput.
- Add `--help` through a shared CLI parser; `profile_decode` currently treats
  unknown arguments as a panic.
