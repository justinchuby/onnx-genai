## Design в†’ Implementation Gap Analysis (2026-07-14, Tyrell)

Evidence-based audit of `docs/DESIGN.md` (engine) + `docs/ORT2.md` (nxrt runtime) against the actual workspace crates. Verified by inspecting registrations/types/tests вЂ” **not** by trusting the prose above. Legend: вњ… implemented + tested В· рџџЎ partial/stubbed В· вќЊ design-only / not started. Ordered roughly by roadmap impact.

### Runtime (nxrt) вЂ” top gaps

1. **Op coverage (ep-cpu) вЂ” рџџЎ the #1 conformance blocker.** `onnx-runtime-ep-cpu` registers only **~30 op types** (24 standard `ai.onnx`: Add, Sub, Mul, Div, Pow, Min, Max, Sqrt, Erf, Tanh, MatMul, Gemm, Relu, LayerNormalization, ReduceMean, Softmax, Reshape, Transpose, Gather, Cast, Shape, Slice, Unsqueeze, Expand, Constant, Identity + 6 fused/contrib: FusedMatMulBias, FusedGemm, FusedAttention, Gelu) вЂ” see `crates/onnx-runtime-ep-cpu/src/kernels/mod.rs`. The upstream ONNX opset is ~200 ops; **~90+ core ops are unregistered** (Conv, MaxPool/AveragePool/GlobalPool, Sigmoid, Abs, Clip, Concat, Split, Where, Squeeze, Pad, Sigmoid/GELU-variants, ArgMax, TopK, all recurrent RNN/GRU/LSTM, all quantization Q*/DequantizeLinear/MatMulInteger, Resize, etc.). Upstream conformance (`docs/EP_CONFORMANCE.md`): **158 / 1198** dtype-opset pytest cases pass (1038 fail, 2 skip); the Rust runner covers 25 PASS / 3 UNSUPPORTED. **This gap gates real-model coverage and the whole conformance story.**

2. **Kernel dtype coverage (ep-cpu) вЂ” рџџЎ f32-only compute.** `kernels/mod.rs::to_dense_f32` calls `require_dtype(..., Float32)` вЂ” every compute kernel rejects non-f32. Only `Identity` handles bf16/f16/i32 bit-exact (`kernels/identity.rs`). **No fp16/bf16/int8 compute kernels exist**, so quantized and half-precision models cannot run on the CPU EP even for registered ops. (IR/loader/`Value` model bf16/f16/fp8/int4 correctly вЂ” the gap is purely in kernels.) Unblocks quantized-model milestone alongside op coverage.

3. **CUDA EP Phase 2b вЂ” рџџЎ minimal GPU slice.** `onnx-runtime-ep-cuda` registers only **`MatMul` (cuBLASLt, true fp32/f16/bf16) + a naive `Attention` SDPA baseline** (`kernels/mod.rs::CUDA_PHASE2A_OPS`); the SDPA kernel is **f32-only** and explicitly *not* FlashAttention (`kernels/attention.rs`). Deferred (clear `// Phase 2b:` markers): cuDNN-fused SDPA в†’ **FlashAttention-3**, fused LayerNorm/RMSNorm/RoPE/softmax via CuTe, NVRTC elementwise, **FP8**, fused bias/act epilogues, strided/broadcast batched GEMM, per-stream workspace pooling, true async-copy fences, **CUDA-graph capture**, and **paged-KV В§13.4**. Blocks H200 perf and device-resident serving.

4. **Tracer executor wiring / MissedFastPath real emission sites вЂ” рџџЎ seams only.** `onnx-runtime-tracer` (Chrome/Perfetto export, AutoDiagnosis В§46.6, Roofline В§49.5, `IssueCategory::MissedFastPath`, CUPTI/ITT collectors) is fully built **but not imported by any runtime crate** вЂ” `grep onnx-runtime-tracer crates/*/Cargo.toml` matches only the tracer's own files. No `Executor`/session instrumentation, no В§46 execution-trace log, no `tracing`-crate `Layer` bridge (В§17.1). The `MissedFastPath`/Roofline diagnoses run only on synthetic test spans; **there are no real kernel emission sites feeding them.**

5. **Optimizer passes beyond fusion вЂ” рџџЎ/вќЊ.** `onnx-runtime-optimizer` ships **ConstantFolding вњ… / DeadNodeElimination вњ… / OpFusion вњ…** (LayerNorm, FusedMatMulBias, FusedGemm, FusedAttention, Gelu вЂ” all with CPU kernels + shape rules + bert_toy parity). **`LayoutPropagation`, `PlacementOptimizer`, and cost-model-driven device placement are design-only** вЂ” named but commented out in `optimizer/src/lib.rs::default_passes()`.

6. **Eager execution (`docs/EAGER.md`) вЂ” рџџЎ core dispatch only.** `onnx-runtime-eager` implements real single-op dispatch by building ephemeral IR nodes, resolving kernels through the ep-cpu `ExecutionProvider` and shapes through `onnx-runtime-shape-inference` (`src/dispatch.rs`). It is therefore **bounded by ep-cpu's ~30 ops / f32-only** limits, the В§9.2 kernel-shape fallback is deferred, it is a **standalone crate not wired into any Python surface**, and custom-domain / subgraph / autograd features are unbuilt.

7. **EPContext В§55 вЂ” вњ… COMPLETE (recorded for accuracy).** Produce в†’ dump (`*_ctx.onnx`, embed/external) в†’ reload в†’ consume + `ep.context_*` C-API options are all landed and model-agnostic (`onnx-runtime-loader/src/{epcontext,writer,encoder}.rs`, `session/src/epcontext.rs`, capi). No CPU-EP compile step exercises it yet (mock-EP tested); real auto-export awaits a compiling EP.

### Engine (onnx-genai) вЂ” memory / serving layer (the largest design-only surface)

8. **`KvCacheManager` abstraction вЂ” вќЊ not realized as designed.** DESIGN В§3.2/В§decode-loop centre everything on a `KvCacheManager` (page-table mapping + append). **No such type exists** (`grep KvCacheManager crates/` в†’ nothing); the engine composes `PageTable` + `PrefixCache` + `PagedCache` (`onnx-genai-kv`) directly.
