### 2026-07-23: Validate If branch signatures before EP optimization
**By:** Sidle
**What:** Added recursive static validation of `If` branch output counts and dtypes at the start of `Executor::build`, before CPU/CUDA EP graph passes. The runtime `If` path reuses the same validator. Updated the two negative control-flow tests to assert build-time `SessionError::ControlFlow`, and added a valid `If` regression whose two branches contain fuseable `MatMulNBits` + `Add` patterns and execute both branches correctly.
**Why:** The failures are pre-existing on `perf/cpu-ep-mlas` at `7d9364e`. Baseline ran 0/2 requested negative tests: the count mismatch was masked by `CpuMatMulNBitsBiasFusion`'s debug postcondition as `MissingProducer(ValueId(3))`, because recursive `Graph::validate` treated a branch output that captures an outer value as producer-less; the dtype mismatch was instead rejected by post-pass shape inference with a generic `ShapeInfer` error. Static control-flow signature validation is the earliest layer with enough parent/branch context to emit the intended diagnostics, and it prevents malformed branches from reaching EP optimization without disabling fusion for valid child graphs. After the fix, `control_flow` is 20/20 (including the new fusion regression), session lib tests are 64/64, and CPU EP MLAS tests are 817 passed + 3 ignored with 10/10 numeric regressions. The full session command also reached the unrelated `executor` integration suite and reported 2 failures; both reproduce unchanged on the base commit (`from_graph_rejects_cyclic_execution_plan` and `unsupported_op_error_formats_unnamed_node_gracefully`).
