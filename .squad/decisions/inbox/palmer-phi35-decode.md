### 2026-07-23: Phi-3.5 decode gap is not a MatMulNBits fallback
**By:** Palmer
**What:** Reproduced and profiled Phi-3.5-mini int4 decode. I did not land a kernel change because the proposed root cause is false and the only MLAS win measured was too small/noisy to justify changing the numerics-sensitive default.

- Clean native reproduction (128 tokens, load 4.04): **14.66 tok/s**. The independently supplied clean load-gated ORT result is **21.93 tok/s** (1.50x faster than this reproduction; the original A/B was 13.59 vs 21.93, 1.61x).
- Clean-start native op profile (load 4.79, 14 steady decode steps): **51.934 ms/step** instrumented; `If` **34.075 ms / 65.61%**, `MatMulNBits` **13.095 ms / 25.22%**, `GroupQueryAttention` **2.944 ms / 5.67%**, SkipSimplifiedLayerNormalization 1.38%, SiLU 0.95%, Mul 0.74%.
- Model inspection found **161/161** `MatMulNBits` nodes are bits=4, block_size=32, accuracy_level=4, three inputs, with no zero point or `g_idx`. On this AVX512-VNNI host, decode therefore enters the packed-int4 VNNI path at `matmul_nbits.rs:376-416`; it does **not** dequantize weights to f32. `try_mlas_sqnbit` deliberately returns before packing for small-M accuracy-level-4 at `matmul_nbits.rs:553-560`.
- Forced MLAS CompInt8 probe (128 tokens, load 4.53) produced identical token IDs and **14.87 tok/s**, versus the nearest clean default-native trial **14.47 tok/s** at load 4.67 (+2.8%). This does not close the gap and is too close to run-to-run/shared-host noise to flip the general default safely.
- A general consecutive constant-`If` output cache was prototyped because Phi's RoPE-cache `If` has two Constant-only outputs `[4096,48]`. It did not improve clean throughput (14.47 vs 14.66 tok/s), so it was reverted rather than shipping unproven complexity. The profiler's large `If` bucket is therefore substantially scheduling/instrumentation time, not 34 ms of constant copying.
- CPU EP validation: `cargo test -p onnx-runtime-ep-cpu --features mlas --quiet` — **815 passed, 3 ignored**, integration **10 passed**, doctest **1 ignored**.
- Qwen2.5-0.5B and Qwen2.5-Coder-7B no-regression reruns could not be admitted: repeated waits for load <=5 were interrupted by concurrent 96-thread benchmark jobs immediately after the gate. No production source change was retained, so the existing Qwen wins are unchanged by this diagnostic commit.

**Why:** The safe next step is to profile below the executor's wall-clock attribution. Add low-overhead phase counters around `exec_if`/`run_subgraph` and persistent-SPMD dispatcher wait, then compare persistent pool, flat pool, and forced MLAS under an exclusive load gate. If repeated isolated trials confirm MLAS CompInt8 wins across block-32 accuracy-level-4 shapes without token/output drift, route it through a topology/shape crossover; otherwise target the non-MatMul dispatcher/control-flow stall. The current evidence does not support a dequant-f32 fallback fix.
